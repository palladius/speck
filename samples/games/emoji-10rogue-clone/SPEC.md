# Emoji 10Rogue: Browser Coffee-Break Roguelike

## Problem Statement\n\nClassic roguelikes provide deep, tactical play but often suffer from high barriers to entry due to complex interfaces, archaic keyboard-only controls, and intimidating ASCII graphics. Native PC executables also limit accessibility for casual players who prefer quick gaming sessions on mobile browsers. There is a need for a modern, zero-install, web-friendly clone of the classic \"10rogue\" game that preserves its swift turn-based gameplay, progressive dungeon crawl, and atmospheric light-as-you-explore fog of war, while delivering approachable emoji-based visuals and dual keyboard/mobile control schemes.\n\n## Goals\n\n* **Pure Browser Playability**: Run purely in any modern browser with zero downloads, installs, or external compilation steps.\n* **Vibrant Emoji Aesthetics**: Use colorful, universally recognizable emoji characters for the player, monsters (rats, trolls, dragons), items (potions, swords), and environments.\n* **Atmospheric Fog of War**: Implement a responsive dynamic lighting system where map areas start completely dark and illuminate as the player explores, leaving a dimmed memory of visited rooms.\n* **Balanced 10-Level Crawl**: Feature 10 progressively difficult procedurally generated floors, scaling monster attributes, and a clear victory condition (beating the boss on Floor 10).\n* **Dual Input Schemes**: Fully controllable via keyboard (WASD/Arrows) for desktop players, and via on-screen virtual directional buttons for mobile/touch screen players.\n\n## Non-Goals\n\n* **Multiplayer or Server Persistence**: The game will be entirely local/single-player. Online leaderboards or multiplayer lobbies are out of scope.\n* **Complex Inventory Screens**: Keep item interaction immediate and simple (e.g., step on a potion to heal, step on a sword to automatically upgrade attack value) rather than introducing nested inventory menus.\n* **Sound FX / Music Synthesis**: Audio design is excluded from the initial scope to prioritize lightweight, rapid visual and mechanics implementation.\n\n## Technical Plan / Approach\n\n* **Architecture**: A lightweight, single-page application built on standard HTML5, CSS3, and modern Vanilla JavaScript.\n* **Rendering Pipeline**: HTML5 Canvas will be used over DOM grids to ensure snappy rendering performance, clean scaling of emoji sprites, and pixel-precise alpha composite blending for the field-of-view (FOV) shadow overlay.\n* **Grid Logic**: A 2D array representation of the map (e.g., 20x20 or 25x25 grid). Each grid square stores tile type (Wall, Floor, Stairs), fog state (Unexplored, Visited/Shadowed, Visible), and optional entities (Player, Enemy, Item).\n* **Field of View (FOV)**: A simple ray-casting or shadow-casting algorithm computed relative to the player's position on every turn. The canvas will render fully lit cells normally, visited-but-unseen cells with a semi-transparent black overlay, and unexplored cells as solid black.\n* **Control Integration**: Keyboard listeners mapped to directions, alongside a fixed-position CSS overlay grid simulating a touch-friendly virtual D-pad on mobile viewports.\n\n## Alternatives Considered\n\n* **CSS Grid / DOM Element Rendering**: Generating a grid of <div> elements for tiles. While simple to implement, dynamic lighting calculations and opacity transitions across hundreds of DOM nodes cause significant rendering lag on mid-to-low-tier mobile browsers. HTML5 Canvas provides a single drawing target and consistent performance.\n* **Third-Party Engines (rot.js)**: A solid engine for browser roguelikes, but introducing external script dependencies adds load latency and reduces visual customization freedom. Writing a native 80-line shadow-caster and room-generator in pure JS keeps the codebase ultra-lightweight and maintainable.\n\n## Implementation Plan\n\n* **Phase 1: Engine & Map Generation**: Establish the HTML Canvas viewport. Implement a basic BSP (Binary Space Partitioning) or room-and-corridor generator that places the player and floor transitions (stairs).\n* **Phase 2: Movement & Input**: Implement grid-locked movement for the player, supporting both keyboard listeners and an HTML virtual layout for touch-input. Add boundary checking for wall collisions.\n* **Phase 3: Dynamic Lighting**: Write the shadow-casting algorithm. Integrate the three-tier fog system (Unexplored, Shadowed/Visited, Currently Visible) visually rendering onto the canvas.\n* **Phase 4: Enemies & Turn Loop**: Introduce basic monsters with simple chasing AI. Set up the turn-taking queue where enemies execute their move/attack actions after the player moves.\n* **Phase 5: Progression & Combat Mechanics**: Add health pools, combat log, and bumper-style combat (moving into an enemy triggers an attack). Implement stair interactions to load the next level, generating stronger monsters and better items.\n* **Phase 6: Mobile Polish & HUD**: Construct responsive layout adjustments so the Canvas and touch buttons fit beautifully on portrait mobile screens. Add a HUD showing current Floor, HP, and XP status.\n\n## Open Questions\n\n* **Emoji Font Standardizing**: Emojis render differently depending on operating systems (Apple, Android, Windows). Will variations in emoji designs alter hitboxes or readabilities across platforms? (Testing on common mobile devices is required to ensure alignment remains centered inside canvas cells.)\n* **Game Balance**: How steep should the difficulty ramp be per floor to ensure the average run takes exactly 10 minutes to complete?
